How should the game be architected?

This shapes how easy it is to add multiplayer later and how complex the build is now

A

Client-Heavy

Game state lives entirely in the browser. Supabase stores card definitions only. AI runs in-browser. Fast to build, works offline.

Pros

  • Simplest to build
  • No server costs
  • Works offline

Cons

  • Multiplayer requires a full rewrite later
  • Card data exposed in client
B

Server-Authoritative

Game state managed via Supabase Realtime + Next.js API routes from day one. Multiplayer-ready architecture.

Pros

  • Multiplayer drops in cleanly
  • Cheat-proof
  • Consistent state everywhere

Cons

  • Significantly more complex upfront
  • Requires Supabase Realtime setup
C — Recommended

Hybrid (Clean Interface)

Cards & rules live in Supabase. Game state lives in browser now, behind a clean GameEngine interface that can move server-side later without touching UI code.

Pros

  • Fast to build now
  • Multiplayer upgrade path without rewrite
  • Card database ready from day one

Cons

  • Slightly more upfront design
  • Still client-side state (not cheat-proof)
